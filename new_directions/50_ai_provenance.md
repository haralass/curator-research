# 50 — Academic Integrity & AI Provenance: Source Tracking for Personal Files
_Research date: 2026-05-31_

---

## Why This Matters for Curator

As AI-generated content proliferates, a local file organizer encounters a new class of problem: files in a user's collection may be human-authored, AI-generated, derived from AI-assisted work, or scraped/transformed from external sources — and distinguishing these provenance categories is increasingly important. For students, Curator's file collection may contain both original work and AI-generated drafts; for researchers, it may contain papers, notes, and LLM-synthesized summaries; for professionals, it may contain contracts, reports, and AI-generated boilerplate.

The stakes are real. In 2025 a Yale student was suspended based on a false GPTZero detection on their legitimately human-written essay. False provenance attribution by an AI system can have serious consequences. Conversely, the absence of provenance tracking means a user under academic integrity scrutiny has no evidence chain to prove their work is original. Curator, which reads and processes personal files, is uniquely positioned to provide or record provenance metadata — but must do so carefully given the state of detection technology.

Three distinct capabilities are relevant: (1) **C2PA-style technical provenance**, which cryptographically binds content to its origin at creation time; (2) **AI content detection**, which attempts to classify existing content as human or AI-generated post-hoc; and (3) **Curator-native provenance tracking**, which records the history of Curator's own operations on files — what was read, classified, or modified, and when. These three capabilities have very different reliability profiles and appropriate use cases.

## C2PA: Content Provenance Standard

### What C2PA Is

The Coalition for Content Authenticity and Provenance (C2PA, https://c2pa.org/) is a technical standards body co-founded by Adobe, Microsoft, Sony, BBC, Intel, and others. The C2PA specification (v2.2, May 2025) defines a format for embedding cryptographically signed provenance assertions directly into media files. A C2PA "manifest" is a JUMBF (JPEG Universal Metadata Box Format) container holding:
- **Content credentials**: who created the file, with what tool, at what time
- **Ingredient hashes**: cryptographic links to source materials (e.g., "this image was derived from this other image")
- **Actions history**: what operations were performed (cropped, AI-generated, edited)
- A **COSE signature** (CBOR Object Signing and Encryption) with an X.509 certificate chain

The manifest is embedded directly in the file in a way that survives most format conversions. SHA-256 hashes bind the manifest to the specific file content; any modification invalidates the signature.

### Current Adoption and Tooling

Open-source tooling is production-ready:
- **`c2pa-rs`** (Rust): https://github.com/contentauth/c2pa-rs — the reference implementation, used by Adobe and Microsoft in production
- **`c2pa-python`** (Python bindings): https://github.com/contentauth/c2pa-python — direct Python API for reading and writing C2PA manifests
- **`c2patool`**: command-line tool for inspecting and creating manifests

C2PA v2.2 supports: JPEG, PNG, PDF, MP4, MOV, WAV, WebP, AVIF. Plain text and Markdown are not supported until v2.3 (January 2026), which uses a Unicode variation selector approach — fragile and not yet widely supported.

### Applicability to Curator

For supported file types (PDF, image, video), Curator could:
1. **Read existing C2PA manifests** to extract provenance information already embedded by the creating application (Adobe Photoshop, Firefly, Dall-E 3 with C2PA support)
2. **Write C2PA manifests** to record Curator's own processing operations (classified on date X, cluster assigned Y, moved from path A to path B)
3. **Display provenance chains** in the Curator UI — showing the user "this file was AI-generated by Firefly on 2024-11-15"

**Critical limitation:** C2PA requires a trusted certificate authority (CA) to sign manifests. For a consumer app, self-signed certificates work locally but cannot be validated by external parties. Free trusted CAs for C2PA do not exist as of mid-2026; the C2PA trust registry is enterprise/media-company oriented. This means C2PA-based provenance in Curator is locally verifiable but not externally auditable without infrastructure investment.

## AI Content Watermarking

### Kirchenbauer et al. Red-Green Watermarking (2023)

Kirchenbauer, J., Geiping, J., Wen, Y., Kautz, J., Kemker, R., Goldstein, T. "A Watermark for Large Language Models." arXiv:2301.10226. ICML 2023.

The foundational LLM text watermark. At generation time, the LLM's vocabulary is randomly partitioned into "green" and "red" token sets (partitioning changes at each position based on the prior token). The model is steered to preferentially sample green tokens. At detection time, compute the fraction of green tokens in the text; if significantly above 50%, the watermark is present. Statistical test (z-score) provides formal detection confidence.

**Key limitation:** The 2024 BIRA attack (arXiv:2402.19361) achieves **99.5% watermark evasion** by substituting near-synonyms until the green-token ratio drops below detection threshold. The attack requires only a few hundred LLM calls and is practical for adversarial users. This effectively defeats Kirchenbauer-style watermarks for adversarial detection purposes.

### SynthID-Text (Google DeepMind, 2024)

Dathathri, S., et al. "Scalable watermarking for identifying large language model outputs." *Nature*, 2024. Open-sourced October 2024 at https://github.com/google-deepmind/synthid-text.

SynthID-Text uses "Tournament Sampling" — a more sophisticated token sampling procedure that embeds a watermark that is more robust to paraphrasing attacks than Kirchenbauer's method. Deployed at scale in Google's Gemini models.

**Critical limitation:** SynthID-Text can only be embedded at generation time by the LLM provider. Curator can detect SynthID-Text watermarks in text generated by Gemini, but only if Google provides a public detection API. For files generated by local LLMs (Ollama, llama.cpp), SynthID-Text is not applicable — there is no mechanism to embed it. Curator could embed its own watermark in AI-generated content it produces (e.g., AI-generated file rename suggestions, AI-generated summaries), but cannot detect watermarks in content generated by external LLMs without specific cooperation from those providers.

## AI Text Detection

### DetectGPT — Mitchell et al. (2023)

Mitchell, E., Lee, Y., Khazatsky, A., Manning, C.D., Finn, C. "DetectGPT: Zero-Shot Machine-Generated Text Detection Using Probability Curvature." arXiv:2301.11305. ICML 2023.

DetectGPT exploits the observation that LLM-generated text tends to lie in local maxima of the log-probability space — random perturbations of LLM-generated text tend to have lower log-probability, while human-written text shows less consistent curvature. The method requires white-box access to the generator model's log-probabilities, which limits practical applicability.

### Commercial Detectors: GPTZero, ZeroGPT, OpenAI Classifier

The empirical reliability data for commercial detectors is damning:
- **GPTZero:** Claims 0.5% false positive rate on its own test set; independent research finds **18–23% false positive rate** on real-world academic writing samples (Walters, 2023; Sadasivan et al., 2023)
- **OpenAI's classifier:** Retired in July 2023 due to "low rate of accuracy"
- **Systematic bias:** A 2024 Stanford study found commercial detectors systematically misclassify non-native English writing as AI-generated — penalizing ESL students disproportionately
- **Adversarial robustness:** Sadasivan et al., "Can AI-Generated Text be Reliably Detected?" (arXiv:2303.11156) proved a *theoretical impossibility result*: as LLM output distributions approach human text distributions, no detector can maintain both low false positive and low false negative rates simultaneously

**Curator policy implication:** Curator must not present AI detection results as definitive. The risk of a false positive (incorrectly flagging a user's legitimate work as AI-generated) is real, documented, and professionally harmful. Detection results should be presented as *advisory signals* with explicit uncertainty, not conclusions.

## W3C PROV and Research Data Provenance Standards

### W3C PROV-DM

The W3C PROV Data Model (https://www.w3.org/TR/prov-dm/) provides a standard vocabulary for describing provenance as a directed graph of three entity types:
- **Entity**: a thing that exists (e.g., a file)
- **Activity**: something that happens (e.g., "Curator classified this file")
- **Agent**: something responsible for an activity (e.g., Curator, the user, an LLM)

PROV-DM has the right semantic model for Curator's provenance needs: `wasGeneratedBy`, `wasDerivedFrom`, `wasAttributedTo`, `wasAssociatedWith`. However, PROV-DM has no standard file embedding mechanism — it is an abstract data model, not a file format standard.

### DataCite Metadata Schema 4.6

DataCite 4.6 (the standard for research data citation, used by Zenodo and institutional repositories) has no formal AI-authorship field as of mid-2026. Proposals for an `AIAuthor` or `AIContributor` role are under discussion in the DataCite working group. Dublin Core can carry AI creator names via `dc:creator` informally.

### Gap

All three standards have gaps in handling AI-generated or AI-assisted content. A Curator-native JSON-LD sidecar format would need to bridge them:

```json
{
  "@context": "https://www.w3.org/ns/prov#",
  "@type": "Entity",
  "prov:wasAttributedTo": {
    "@type": "prov:Agent",
    "prov:type": "Human",
    "schema:name": "User"
  },
  "curator:aiProcessingHistory": [
    {
      "activity": "classified",
      "agent": "Curator 1.0 / nomic-embed-text",
      "timestamp": "2025-11-15T14:32:00Z",
      "result": "University/Physics/Lecture Notes"
    }
  ],
  "curator:contentHash": "sha256:a3f5..."
}
```

## macOS Native Provenance: xattr Capabilities

### com.apple.quarantine

Written by macOS when any file is downloaded via a browser or mail client. Contains: the originating application bundle ID, the download date, and a quarantine event UUID linking to `~/Library/Preferences/com.apple.LaunchServices.QuarantineEventsV2` (an SQLite database containing the original download URL).

This is the richest native provenance signal for downloads. Curator can read `kMDItemWhereFroms` (an array of URLs) via `xattr -p com.apple.metadata:kMDItemWhereFroms` or via the Spotlight MDItem API, without any special entitlements.

### Custom xattr Namespace

Curator can write custom xattrs under `com.curator.*`. Recommended policy (informed by Threat 3 in file 49):
- **Do write:** `com.curator.contentHash` (SHA-256 of file content at time of processing — enables detecting modifications)
- **Do not write:** classification labels, cluster IDs, confidence scores, or any AI-derived semantic metadata — these travel with the file on AirDrop/copy and expose the AI's assessments to unintended recipients

## Evidence Chain Architecture for Curator

A Curator-native provenance system should use three layers:

**Layer 1: Content Integrity Hash**
On first processing, compute SHA-256 of file content and record in both the encrypted SQLite database and as `com.curator.contentHash` xattr. Enables detecting file modifications between Curator runs. Provides the anchor for all subsequent provenance claims ("this provenance applies to the version with hash X").

**Layer 2: JSON-LD Provenance Record (W3C PROV-inspired)**
Stored as an encrypted record in the SQLite database (not as a sidecar file, to avoid leakage). Contains: original file location, creation timestamp, all Curator operations (classify, cluster-assign, rename, move), with timestamps and model versions. This is the internal audit trail.

**Layer 3: C2PA Manifest (for supported file types)**
For PDF, JPEG, PNG, MP4 files where the user explicitly requests a provenance certificate (e.g., for submission to an academic journal or employer), Curator generates a C2PA manifest recording: "this file was present in the user's collection on date X, with SHA-256 hash Y, and was not modified by Curator." Uses a locally generated self-signed certificate (locally verifiable but not externally trusted without further infrastructure).

**Append-Only Signed Audit Log**
All Curator operations are written to an append-only JSONL log, with each entry signed by a locally held ECDSA key (stored in macOS Keychain). This provides: (1) a tamper-evident record of everything Curator did, and (2) evidence for a user to demonstrate "my file was in this state on this date, and Curator did not modify it." The log can be exported as a verifiable bundle (JSONL + public key + signature chain).

## Open Problems

**1. No free trusted C2PA CA.** Without an externally trusted certificate, C2PA manifests from Curator are only locally verifiable. An academic institution could run their own C2PA CA for students, but this infrastructure doesn't exist.

**2. No standard for hybrid human+AI content.** If a student writes 80% of an essay and uses an LLM to improve the other 20%, no existing standard represents this mixed provenance. C2PA's "AI-generated" flag is binary.

**3. Retroactive provenance is unsolvable.** For files that existed before Curator was installed, there is no provenance record. Curator can only attest to what happened under its watch. Historical provenance for pre-existing files is unavailable by construction.

**4. xattr stripping on transfer.** The `com.curator.contentHash` xattr will be stripped on email attachment and web upload. Provenance metadata stored in xattrs alone is not reliably portable.

**5. Adversarial watermark evasion is trivially cheap.** BIRA (arXiv:2402.19361) defeated Kirchenbauer watermarking with 99.5% success at minimal cost. Any watermarking scheme Curator could embed in AI-generated text would face the same adversarial threat.
